# 图像显示路径与 DDR3 说明

## 当前结论

当前项目已经从“统计型闭环”升级为“图像预览 + 统计告警闭环”：

```text
OV2640 -> FPGA Capture -> DDR3 Framebuffer -> RGB565 UDP Preview -> Raspberry Pi 5 -> BMP/Web Preview
```

同时保留统计链路：

```text
OV2640 -> FPGA Capture -> ROI/Bright Stats -> Telemetry UDP -> Raspberry Pi 5 -> /api/status
```

## 为什么引入 DDR3

原始摄像头数据是连续流。直接边采边发可以工作，但容易受 UDP 发送节奏、网络忙、分片调度影响。DDR3 帧缓存提供了一个整帧缓冲层，使系统更像正式视觉前端：

- 摄像头写入和网络读出解耦。
- 可以保存最近一帧用于低帧率预览。
- 后续可扩展 ROI 框叠加、冻结画面、抓拍、回放。
- DDR3 本身也成为 FPGA 项目的重要工程卖点。

## 当前图像格式

FPGA 回传：

- 分辨率：`800x600`
- 原始格式：`RGB565`
- 预览分片：UDP payload 携带 RGB565 数据块

Linux 展示：

- C++ 服务重组最近一帧。
- 将 RGB565 转为 BMP。
- Web 页面通过 `/api/preview` 显示。

状态接口中可观察：

- `preview_available=true`
- `preview_width=800`
- `preview_height=600`
- `preview_format=rgb565_bmp`
- `preview_frame_id` 持续增长

## DDR3 与统计链路的关系

DDR3 主要服务图像预览和帧缓存。ROI/亮点统计仍然走实时像素流，不依赖 Linux 对图像做后处理。

这样对面试更有价值：

- FPGA 负责实时路径和轻量预处理。
- DDR3 负责帧级缓存。
- Linux 负责服务化和展示。

## 当前性能定位

当前目标不是高帧率视频流，而是“巡检节点预览 + 状态遥测”：

- 预览画面用于确认场景、框选 ROI 和演示告警。
- 统计遥测用于告警判断和状态闭环。
- Web 页面需要稳定和可解释，不追求摄像头视频播放器级帧率。

## 后续可扩展方向

优先级从高到低：

1. 事件日志导出：把告警事件表导出为 CSV/JSON，方便做巡检记录。
2. 预览降采样：降低网络和 Web 压力，提高显示流畅度。
3. 轻量规则库：支持多组阈值和不同 ROI 的规则配置。
4. 轻量模型推理：如果时间充足，再考虑 Linux 侧或 FPGA 侧简单推理。

## 不建议继续扩大的方向

- 不建议做复杂云端平台。
- 不建议引入完整深度学习训练链路。
- 不建议同时接入多路传感器。
- 不建议为了帧率推翻当前稳定闭环。

当前版本已经足够支撑“FPGA + Linux 边缘智能巡检感知节点”的项目定位。
