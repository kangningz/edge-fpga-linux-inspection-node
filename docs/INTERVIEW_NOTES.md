# 面试讲解要点

## 项目一句话

我做了一个基于 FPGA 和 Linux 的边缘智能巡检感知节点。OV2640 摄像头输入到 FPGA 后，FPGA 完成采集、帧缓存、ROI/亮点统计和 UDP 回传；树莓派 5 上运行 C++ 服务，负责网络通信、参数控制、日志和 Web 展示，最终形成采集、处理、传输、展示、配置和告警的闭环。

## 为什么这个项目不是课程小实验

- 输入是真实摄像头，不是仿真数据。
- FPGA 侧处理了 DVP 时序、SCCB 初始化、跨时钟域、DDR3、UDP 和实时统计。
- Linux 侧不是 Python 脚本，而是 C++ 多线程服务。
- 系统有 Web 控制台、运行参数、日志、状态 API 和 systemd 部署。
- 支持在线修改 ROI 和阈值，能触发告警和蜂鸣器。

## FPGA 侧可以展开讲

- OV2640 初始化：SCCB 写寄存器表，配置 SVGA/RGB565。
- DVP 采集：使用 `PCLK / VSYNC / HREF / D[7:0]` 识别帧、行、像素。
- RGB565 组包：两个 8-bit 数据合成一个 16-bit 像素。
- DDR3 帧缓存：写入整帧数据，再读出做 UDP 预览分片。
- 统计链路：按 ROI 判断像素是否在窗口内，累计亮度和亮点数。
- 跨时钟域：相机域到网络域使用异步 FIFO，单比特事件用同步寄存器或脉冲同步。
- 命令接收：RGMII RX 解 UDP 包，解析命令后写寄存器组。
- 告警逻辑：每帧结束时根据 `bright_count >= alarm_count_threshold` 判断。

## Linux C++ 侧可以展开讲

- `rx_loop`：UDP 接收线程，解析遥测和预览分片。
- `http_loop`：HTTP 服务线程，提供 Web 页面和 JSON API。
- `watchdog_loop`：检测离线超时和配置热重载。
- 状态管理：使用 mutex 保护运行状态、预览帧、运行参数镜像。
- 控制 API：`/api/command` 和 `/api/apply_params` 负责下发 UDP 命令。
- Web 展示：实时显示预览图、状态位、ROI 参数、告警状态。
- systemd：服务可开机启动，适合长期运行节点。
- 告警留痕：服务端记录 `alarm_active` 上升沿事件，保存帧号、阈值、亮点数、ROI 和当时预览快照，避免持续告警刷屏。

## 关键问题定位记录

### DDR3 预览图像异常

现象：Web 画面出现横纹、花屏或数据错位。

定位思路：

- 先确认 OV2640 FIFO 版本画面正常，排除摄像头初始化问题。
- 再检查 DDR3 写入和读出路径是否按 RGB565 字节序一致。
- 最后检查 AXI/MIG 写入时是否在 FIFO 数据不足时继续写。

可讲结论：通过保证 AXI burst 数据完整和读写节奏正确，恢复 DDR3 RGB565 预览画面。

### 命令下发无效

现象：Linux 发送 `buzzer_off` 或 `stop_capture`，FPGA 状态不变化。

定位思路：

- UDP TX 正常，但命令 RX 没被 FPGA 解析。
- RGMII RX 采样相位不对，导致接收数据偶发或错误。
- 复用官方/旧工程中的 RX 相位 MMCM 后命令链路恢复。

可讲结论：这是典型的以太网 RGMII 时序问题，不是上层协议问题。

### `bright_count` 超阈值但没有告警

现象：`bright_count > alarm_count_threshold`，但 `alarm_active=0`。

定位思路：

- 增加 `alarm_enable` 状态位。
- 发现之前执行过 `buzzer_off`，关闭了告警使能。
- 重新 `buzzer_on` 后告警闭环正常。

可讲结论：区分 `alarm_enable` 和 `alarm_active`，避免状态语义混淆。

### 蜂鸣器太短

现象：告警发生时蜂鸣器只响一瞬间。

定位思路：

- 原来 `BEEP` 直接跟随 `alarm_active_cam`，单帧或短脉冲不易听到。
- 增加保持计数器，把短告警脉冲拉伸到约 0.5 秒。
- 无源蜂鸣器不能只给恒定电平，需要在保持窗口内输出约 2 kHz 方波，否则只会在电平跳变瞬间响一下。
- `buzzer_off` 和 `clear_error` 仍可立即清除。

可讲结论：这是工程演示体验优化，不影响采集、DDR3 或 UDP 主链路。

## 面试时避免夸大的点

- 不要说已经做了深度学习模型推理。
- 不要说做了复杂工业检测算法。
- 不要说有完整云端平台。
- 可以说当前是“规则型轻量边缘检测”，后续可接入轻量模型。
